# Frontend OpenAPI alignment: M6 papers management, embed config, QA calibration handlers

This change wires the frontend to five new M6 API endpoints (`sync-qdrant`, `qdrant/paper` DELETE, `mark-inaccessible`, `export-links`, `papers`), adds an embed configuration form to the settings modal, and handles two new QA calibration statuses (`M7_STEP3_QA_CALIBRATION_WAITING_APPROVAL` and `M7_STEP3_QA_CALIBRATION_LOW_KAPPA`). The `apiFetch` wrapper was extended with FormData detection and a `responseType: 'text'` path for the CSV export. The M6 UI provides both an inline papers table with per-row actions and a full-screen modal view.

**Watch for:** The QA calibration retry button targets status `M7_STEP3_QA_CALIBRATION` instead of the spec's `M7_STEP3_QA_CALIBRATION_RETRY` (confirmed). Paper titles from the API are interpolated into innerHTML without escaping, creating an XSS vector if a title contains HTML (confirmed). A dead-code line in `apiFetch` assigns the same value in both ternary branches (confirmed).

## High-level view

A dead-code line (`const body = isFormData ? options.body : options.body`) slipped into the `apiFetch` refactor â€” both branches evaluate to the same expression, so it has no runtime effect but should be removed.

The QA calibration LOW_KAPPA handler's "Retry Calibration" button revises with `target_status: 'M7_STEP3_QA_CALIBRATION'`, but the spec's state machine expects transition to `M7_STEP3_QA_CALIBRATION_RETRY`. If the backend validates target status values strictly, the retry action will fail silently.

The new inline papers table inserts `p.title` directly into `innerHTML` without escaping HTML entities. The modal table (partially pre-existing) has the same pattern, but the inline table is entirely new code in this diff.

The renderer.js import-data upload uses raw `fetch()` with a manually-added `Authorization` header (fixing a pre-existing auth bug), but `API.importData` in api.js now handles the same endpoint correctly with FormData. Two code paths for the same upload means future changes need updating in both places.

## QA calibration state machine mismatch

The `M7_STEP3_QA_CALIBRATION_LOW_KAPPA` handler (renderer.js ~line 1559) calls:

```js
await API.reviseStep(session.id, 'Retry calibration with refined anchors', 'M7_STEP3_QA_CALIBRATION');
```

The OpenAPI spec's state diagram shows `M7_STEP3_QA_CALIBRATION_LOW_KAPPA` transitions to `M7_STEP3_QA_CALIBRATION_RETRY` (not back to `M7_STEP3_QA_CALIBRATION` directly). If the backend's revise handler validates the target status string against a whitelist, the button silently fails. The "Force Proceed" button uses `handleApproval({})` which hits the approve endpoint and should transition to `M7_STEP3_QA_CALIBRATION_FORCE_PROCEED` per the spec's approve behavior from this state.

## XSS in paper title rendering

Both the inline table and the modal table insert `p.title` directly into template literals assigned to `innerHTML`:

```js
rows += `<td ...>${p.title || '-'}</td>`;
```

The `title` attribute on hover is escaped with `.replace(/"/g, '&quot;')`, but the cell content itself is not escaped. If the backend returns a paper title like `<img src=x onerror=alert(1)>`, it executes. The inline table is new code in this diff that replicates a pre-existing vulnerability from the modal. At minimum, the new inline table rows should escape `<`, `>`, and `&` before interpolation.

## Dual import-data code paths

The renderer.js still has a raw `fetch()` call for `/sessions/${session.id}/import-data` (around line 750). This PR adds an `Authorization` header to fix an auth bug, but `API.importData` in api.js already provides the same functionality with automatic FormData handling and auth injection via `apiFetch`. The raw call has its own error-handling path (checking `res.ok`, parsing JSON manually) while `API.importData` uses the centralized error handling in `apiFetch`. Future changes to auth or error handling would need updating in both places.
